## STEP 2 — Create the JWT company_id helper function

This function extracts the company_id from the authenticated JWT token so RLS policies can use it:

```sql
CREATE OR REPLACE FUNCTION auth.company_id()
RETURNS integer AS $$
  SELECT NULLIF(current_setting('request.jwt.claims', true)::json->>'company_id', '')::integer;
$$ LANGUAGE sql STABLE;
```

If the JWT stores company_id as a string or UUID instead of integer, adjust the return type accordingly. Check the JWT structure in the codebase to confirm the exact field name and type used for company_id before creating this function.

## STEP 3 — Create RLS policies for every table

Apply the following policy pattern to every table enabled in Step 1. The pattern gives authenticated users access only to rows matching their company_id, and gives the service role (used by the backend) unrestricted access:

```sql
-- Template for each table (replace TABLE_NAME with actual table name):

-- Policy 1: Authenticated users see only their company's data
CREATE POLICY "tenant_isolation_select" ON TABLE_NAME
  FOR SELECT
  USING (company_id = auth.company_id());

-- Policy 2: Authenticated users can only insert their own company's data  
CREATE POLICY "tenant_isolation_insert" ON TABLE_NAME
  FOR INSERT
  WITH CHECK (company_id = auth.company_id());

-- Policy 3: Authenticated users can only update their own company's data
CREATE POLICY "tenant_isolation_update" ON TABLE_NAME
  FOR UPDATE
  USING (company_id = auth.company_id())
  WITH CHECK (company_id = auth.company_id());

-- Policy 4: Authenticated users can only delete their own company's data
CREATE POLICY "tenant_isolation_delete" ON TABLE_NAME
  FOR DELETE
  USING (company_id = auth.company_id());

-- Policy 5: Service role bypass — backend API has full access
CREATE POLICY "service_role_bypass" ON TABLE_NAME
  FOR ALL
  USING (auth.role() = 'service_role');
```

Apply this pattern to every table listed in Step 1. Run them all as a single SQL transaction so if any fail, none are applied:

```sql
BEGIN;
-- paste all CREATE POLICY statements here
COMMIT;
```

## STEP 4 — Verify policies are working

Run these verification tests after applying all policies:

**Test 1 — Tenant isolation:**
Using a test JWT for a non-Phes tenant, attempt to SELECT from the clients table. Confirm zero rows are returned even though Phes has 1,231+ client records.

**Test 2 — Own data accessible:**
Using the Phes JWT, SELECT from clients. Confirm Phes client records are returned correctly.

**Test 3 — Insert isolation:**
Attempt to INSERT a client record with a different company_id than the authenticated JWT. Confirm the insert is rejected with a policy violation error.

**Test 4 — Service role bypass:**
Using the service role key (used by the Express backend), confirm full read/write access to all tables. This is required for the backend API to function.

**Test 5 — No broken API routes:**
After applying RLS, open the live app and confirm the following pages load correctly with real data: Dashboard, Dispatch Board, Customers, Employees, Invoices, Payroll, Leads. If any page shows empty data that should have data, report which table's RLS policy is blocking the backend and fix it.

## STEP 5 — Fix any broken queries

If any API route returns empty data after RLS is applied, it means that route is using the anon key instead of the service role key for its database queries. For any broken route:

1. Identify which Supabase client instance it is using
2. Confirm the backend Express API is using the service role key (SUPABASE_SERVICE_ROLE_KEY) for all server-side queries — not the anon key
3. If SUPABASE_SERVICE_ROLE_KEY is not in Replit Secrets, add it now — find it in Supabase dashboard → Settings → API → service_role key
4. Update any broken routes to use the service role client

The anon key should only be used for public-facing endpoints like the booking widget zone check. All authenticated backend routes must use the service role key.
