Here is the full rewritten Phase 2 prompt — all patches baked in, nothing split out:

You are working on CleanOps Pro, a multi-tenant SaaS platform for 
residential and commercial cleaning businesses.

TECH STACK: React 18, Tailwind, Express/Node.js, Supabase (Postgres 
+ RLS), Plus Jakarta Sans (ONLY font).

DESIGN SYSTEM — NON-NEGOTIABLE:
- Background: #F7F6F3
- Cards: #FFFFFF
- Primary text: #1A1917
- Secondary text: #6B6860
- Borders: #E5E2DC
- Accent: var(--brand) — dynamic per tenant
- NO emojis, NO dark mode, light theme only
- Plus Jakarta Sans only
- Status badges: pill shape, muted background, colored text
- Skeleton loaders (not spinners) for all async loads
- Graceful empty states on all list and table views
- All monetary values: USD, 2 decimal places, comma-formatted

All endpoints must be protected by JWT middleware.
All queries must include company_id scoping (RLS enforced).
Never show data across tenant boundaries.

FEATURE 1: INCENTIVE + BONUS TRACKING
═══════════════════════════════════════════════════════

── DATABASE ──

Create incentive_programs table:
{
  id,
  company_id,
  name,
  type              enum: performance / attendance / 
                         retention / referral / custom,
  trigger_metric    text (description of what triggers it),
  threshold_value   decimal (nullable),
  reward_amount     decimal,
  reward_type       enum: cash / gift_card / pto / other,
  monthly_budget_cap decimal (nullable),
  is_active         boolean,
  effective_date    date
}

Create incentive_earned table:
{
  id,
  company_id,
  employee_id,
  program_id,
  earned_date       date,
  amount            decimal,
  notes             text,
  status            enum: pending_approval / approved / 
                         rejected / paid,
  approved_by       employee_id (nullable FK),
  approved_at       timestamp (nullable),
  rejection_note    text (nullable),
  paid_date         date (nullable),
  paid_via_payroll_run_id (nullable FK)
}

── BACKEND ──

CRUD /api/incentives/programs
  — company_id scoped
  — include mtd_awarded computed field on GET 
    (sum of approved + paid incentive_earned 
     for current calendar month per program)
  — if monthly_budget_cap is set, include 
    budget_remaining in response

POST /api/incentives/award
  — manually award an incentive to an employee
  — APPROVAL RULES:
    If awarded by Owner role → status = approved (auto)
    If awarded by Admin role → status = pending_approval
    Notify owner via dashboard alert when 
    pending_approval record is created
  — BUDGET CAP CHECK:
    Before inserting, check mtd_awarded vs 
    monthly_budget_cap for the program
    If cap reached → return 400 with message:
    "Monthly budget cap of $X has been reached 
     for this program"
    If no cap set → proceed without check

GET /api/incentives/earned?employee_id=
  — earnings history for one employee
  — include program name, status, paid_date

GET /api/incentives/pending-approval
  — Owner role only
  — return all pending_approval records 
    across company with employee name, 
    program name, amount, awarded_by, earned_date

POST /api/incentives/:id/approve
  — Owner only
  — sets status = approved, 
    approved_by = current user, approved_at = now

POST /api/incentives/:id/reject
  — Owner only
  — requires rejection_note
  — sets status = rejected

GET /api/incentives/unpaid
  — returns all approved (not yet paid) 
    incentive_earned records
  — feeds into Close Week payroll lock flow

Add "Incentives" section under the Payroll area in

Incentives page has two tabs:

  - Status badge colors:

FEATURE 2: CLIENT SATISFACTION + NPS
═══════════════════════════════════════════════════════

── DATABASE ──

Create satisfaction_survey table:
{
  id,
  company_id,
  job_id,
  customer_id,
  token             uuid unique (for /survey/:token),
  sent_at           timestamp,
  responded_at      timestamp (nullable),
  nps_score         integer 0–10 (nullable),
  rating            integer 1–5 (nullable),
  comment           text (nullable),
  follow_up_required boolean default false,
  suppressed        boolean default false,
  suppressed_reason text (nullable)
}

── BACKEND ──

POST /api/satisfaction/send
  — Trigger: manual or auto on job completion
  — THROTTLE CHECK (run before any send):
    Query satisfaction_survey for this customer_id
    where sent_at > now() - interval '30 days'
    and suppressed = false
    If record found → insert new record with 
    suppressed = true, 
    suppressed_reason = 'throttled — sent within 30 days'
    Return 200 with { sent: false, reason: 'throttled' }
    Do not send email
  — If no throttle hit:
    Generate unique token (uuid)
    Insert satisfaction_survey record
    Send email via Resend with /survey/:token link
    Return 200 with { sent: true }

POST /api/satisfaction/respond (PUBLIC — no auth)
  — Validate token exists and responded_at is null
  — Accept: nps_score, rating, comment
  — Set responded_at = now()
  — Set follow_up_required = true if nps_score <= 6
  — Return 200 confirmation

GET /api/satisfaction/results
  — JWT protected, company_id scoped
  — Return:
    {
      nps_rolling_30d,         integer
      avg_rating_30d,          decimal
      surveys_sent_30d,        integer (suppressed excluded)
      surveys_responded_30d,   integer
      response_rate_pct,       decimal
      follow_up_queue_count,   integer
      nps_by_employee: [
        { employee_id, name, avg_rating, 
          response_count }
      ],
      benchmark: {
        residential: { low: 38, high: 52 },
        commercial:  { low: 28, high: 44 },
        company_type: (from company settings),
        position: 'below' | 'within' | 'above'
      }
    }

GET /api/satisfaction?customer_id=
  — per-client survey history

Branded with the tenant's var(--brand) color

Mobile-first layout — optimized for phone.
