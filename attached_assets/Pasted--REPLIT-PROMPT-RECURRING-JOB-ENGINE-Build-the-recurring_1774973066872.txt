**STEP 2 — ENGINE LOGIC**

Build a function `generateRecurringJobs(companyId, daysAhead = 60)` that:

1. Fetches all `recurring_schedules` where `status = 'active'` and `company_id = companyId`

2. For each schedule, determines the next occurrence dates based on frequency:
   - `weekly` — every 7 days from the anchor date
   - `biweekly` — every 14 days from the anchor date
   - `every_4_weeks` — every 28 days from the anchor date
   - `monthly` — same day of month each month

3. The anchor date logic:
   - If the schedule has a `next_scheduled_date` column, use that
   - If not, use the schedule's `created_at` date as the anchor, then calculate forward to the next future occurrence from today
   - Never generate a job for a date in the past

4. For each occurrence date within the next `daysAhead` days:
   - Check if a job already exists for this `recurring_schedule_id` on that date — if yes, skip it (no duplicates)
   - If no job exists, create a job record with:
     - `company_id` from the schedule
     - `client_id` from the schedule
     - `recurring_schedule_id` linking back to the schedule
     - `scheduled_date` = the calculated occurrence date
     - `scheduled_time` from the schedule
     - `scope` from the schedule
     - `service_address` from the schedule
     - `status` = 'scheduled'
     - `booking_location` = derive from the service address zip code — check `service_zones` table for the matching zone and its location field
     - `created_at` = NOW()

5. After creating the job record, copy technician assignments:
   - Read all rows from `recurring_schedule_technicians` where `recurring_schedule_id` matches
   - Insert a row into `job_technicians` for each technician, linking to the new job_id
   - For schedules with no technician assigned (9 exist), create the job with status = 'scheduled' and leave technicians unassigned — do not skip these

6. Return a summary: `{ jobs_created: N, schedules_processed: N, skipped_duplicates: N, unassigned_jobs: N }`

**STEP 3 — CRON JOB**

Add this to the nightly cron that already runs for rate locks and follow-up sequences:

```javascript
// Run at 2:00 AM nightly
// Generate jobs 60 days out for all active companies
const companies = await db.select().from(companiesTable)
  .where(eq(companiesTable.is_active, true));

for (const company of companies) {
  await generateRecurringJobs(company.id, 60);
}
```

The window is rolling — every night it looks 60 days ahead and fills in any missing jobs. If a job was already created it gets skipped. This means the cron is safe to run multiple times without creating duplicates.

**STEP 5 — RESCHEDULE HANDLING**

When a job is rescheduled (its `scheduled_date` is changed manually), do NOT touch the parent `recurring_schedule`. The schedule continues generating future jobs on the original cadence. The rescheduled job is a one-time exception — it keeps its `recurring_schedule_id` but its date is now independent.

When a job is cancelled:
- If `cancel_this_job_only` — mark that job `status = 'cancelled'`, leave the schedule active
- If `cancel_recurring_service` — mark that job cancelled AND set `recurring_schedules.status = 'inactive'` for that schedule. The engine will no longer generate future jobs for an inactive schedule.

Add these two cancellation options to the job detail page cancel button — a dropdown with "Cancel this visit only" and "Cancel recurring service."
